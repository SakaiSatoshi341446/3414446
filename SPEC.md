# 販売データ分析ダッシュボード 仕様書

## 1. 概要

販売データを可視化するシンプルなダッシュボードを作成する。
バックエンドは Python + FastAPI、フロントエンドは React + TypeScript で構成する。
データベースや外部サービスは使わず、アプリ内で扱うサンプルデータを利用する。

本仕様は、最小構成で動作するダッシュボードを対象とする。作り込みは行わず、基本的な集計表示と簡単な絞り込みに限定する。

## 2. 目的

- 販売データの基本的な見える化を行う
- フロントエンドとバックエンドの疎通が確認できる構成にする
- 少量のサンプルデータを用いて、集計・グラフ表示・フィルタの基本を学べる題材にする

## 3. スコープ

### 対象

- 売上合計の表示
- 注文件数の表示
- 商品別売上の集計表示
- 日別売上の集計表示
- 最小限のフィルタ機能
- サンプルデータの提供

### 対象外

- 認証、認可
- ユーザー管理
- 本番向けのデータベース接続
- リアルタイム更新
- 高度な分析機能
- 複雑な画面遷移

## 4. 想定利用者

- 開発演習の参加者
- ダッシュボード実装のサンプルを確認したい学習者

## 5. システム構成

### バックエンド

- Python
- FastAPI
- サンプル販売データを読み込み、集計済みデータを API として返す

### フロントエンド

- React
- TypeScript
- Vite
- API から取得したデータをカード、表、グラフで表示する

## 6. データ仕様

### サンプルデータ方針

- データはアプリ内の固定サンプルデータを使用する
- 形式は JSON または CSV のどちらでもよい
- 件数は 30 件から 100 件程度を想定する
- 複数日付、複数商品、複数顧客が含まれること

### 販売データ項目

| 項目名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | string | 必須 | 販売データの識別子 |
| date | string | 必須 | 売上日。ISO 形式の日付文字列 |
| product_name | string | 必須 | 商品名 |
| category | string | 任意 | 商品カテゴリ |
| quantity | number | 必須 | 販売数量 |
| unit_price | number | 必須 | 単価 |
| customer_name | string | 任意 | 顧客名 |

### 算出項目

- 売上金額 = quantity × unit_price

## 7. 機能要件

### 7.1 ダッシュボード表示

初期表示時に以下を表示する。

- 売上合計
- 注文件数
- 商品別売上
- 日別売上

### KPI カード

以下の 2 つをカード表示する。

- 売上合計
- 注文件数

### 商品別集計

- 商品ごとの売上合計を表示する
- 表示形式は棒グラフまたは表のどちらかでよい
- 最小構成として、上位 5 件から 10 件程度の表示でよい

### 日別集計

- 日ごとの売上合計を表示する
- 表示形式は折れ線グラフまたは棒グラフでよい
- 対象期間はサンプルデータに含まれる範囲とする

### 7.2 フィルタ機能

以下の簡易フィルタを提供する。

- 日付範囲
- 商品名

条件変更時、ダッシュボード全体の数値とグラフを再取得または再集計して更新する。

フィルタは最小限とし、複数条件の高度な組み合わせや保存機能は不要とする。

### 7.3 明細表示

- フィルタ適用後の販売データ一覧を表で表示する
- 表示項目は date、product_name、quantity、unit_price、売上金額 とする
- ページネーションは不要とし、件数が多い場合もサンプルデータの範囲でそのまま表示する

## 8. API 要件

バックエンドは REST 形式のシンプルな API を提供する。

### 8.1 データ一覧取得

`GET /api/sales`

用途:
- フィルタ後の販売明細一覧を取得する

クエリパラメータ:
- start_date: 開始日、任意
- end_date: 終了日、任意
- product_name: 商品名、任意

レスポンス例:

```json
{
  "items": [
    {
      "id": "S001",
      "date": "2026-06-01",
      "product_name": "Laptop",
      "category": "Electronics",
      "quantity": 2,
      "unit_price": 120000,
      "sales_amount": 240000
    }
  ],
  "count": 1
}
```

### 8.2 サマリー取得

`GET /api/dashboard/summary`

用途:
- KPI を取得する

クエリパラメータ:
- start_date: 開始日、任意
- end_date: 終了日、任意
- product_name: 商品名、任意

レスポンス例:

```json
{
  "total_sales": 1250000,
  "order_count": 18
}
```

### 8.3 商品別集計取得

`GET /api/dashboard/by-product`

用途:
- 商品別売上集計を取得する

クエリパラメータ:
- start_date: 開始日、任意
- end_date: 終了日、任意
- product_name: 商品名、任意

レスポンス例:

```json
{
  "items": [
    {
      "product_name": "Laptop",
      "total_sales": 480000,
      "order_count": 4
    }
  ]
}
```

### 8.4 日別集計取得

`GET /api/dashboard/by-date`

用途:
- 日別売上集計を取得する

クエリパラメータ:
- start_date: 開始日、任意
- end_date: 終了日、任意
- product_name: 商品名、任意

レスポンス例:

```json
{
  "items": [
    {
      "date": "2026-06-01",
      "total_sales": 240000,
      "order_count": 2
    }
  ]
}
```

## 9. 画面要件

画面は 1 ページ構成とする。

### 画面構成

1. 画面タイトル
2. フィルタエリア
3. KPI カードエリア
4. 商品別集計エリア
5. 日別集計エリア
6. 販売明細テーブル

### UI 方針

- シンプルで見やすいレイアウトにする
- フィルタ部品は最小限とする
- グラフは 2 種類までに留める
- レスポンシブ対応は簡易対応とし、PC 画面での閲覧を主対象とする

## 10. 非機能要件

- ローカル環境で起動できること
- サンプルデータで安定して表示できること
- API エラー時にフロントエンドで簡単なエラーメッセージを表示すること
- データ件数が少ない前提のため、性能最適化は最低限でよい

## 11. 開発前提

### バックエンド

- FastAPI で API を実装する
- CORS を有効化し、フロントエンドからのアクセスを許可する
- サンプルデータはファイルまたはメモリ上で保持する

### フロントエンド

- React + TypeScript + Vite で構築する
- API クライアントは fetch ベースのシンプルな実装でよい
- グラフライブラリは 1 つまで導入可とする

## 12. 受け入れ条件

- アプリ起動後、ダッシュボード画面が表示される
- 売上合計と注文件数が表示される
- 商品別集計が表示される
- 日別集計が表示される
- 日付範囲または商品名で絞り込みできる
- 絞り込み結果が KPI、集計、明細に反映される
- サンプルデータのみで動作する
- 画面と API が最低限のエラー処理を持つ

## 13. 実装メモ

- 初期実装では永続化は不要
- サンプルデータは固定値でよい
- バックエンド側で集計して返しても、フロントエンド側で一部加工してもよい
- ただし責務を分かりやすくするため、集計は可能な限りバックエンド側で実施する
